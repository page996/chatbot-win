# 迁移部署方案

本文用于把项目从当前机器迁移到新的 Windows PC，并保证对话、发送、WeFlow、native hook 和控制台状态按当前架构运行。

## 1. 目标机器前置条件

1. Windows 10/11。
2. Python 3.11+，命令行可执行 `py -3` 或 `python`。
3. Node.js + npm，用于启动 `vendor/reference/WeFlow-gitcode`。
4. PC 微信 4.x。当前 native hook 只把 `4.1.10.53` 作为已验证版本；其他版本必须先跑迁移探测，不要直接发送。
5. 如需重新构建 hook，安装 Visual Studio 2022 Build Tools / Community，并包含 C++ x64 toolchain。

## 2. 首次启动顺序

1. 拉取代码后进入仓库根目录。
2. 安装 Python 可选依赖：
   - 基础窗口依赖：`pip install -r requirements-windows-ui.txt`
   - 文档解析：`pip install -r requirements-document.txt`
   - OCR/ASR 按机器选择 `requirements-ocr-light.txt`、`requirements-ocr-gpu.txt`、`requirements-asr-light.txt` 或 `requirements-asr.txt`
3. 配置模型和 WeFlow token：
   - 模型密钥优先在控制台“模型配置/Key pool”录入，或使用配置文件指向环境变量。
   - WeFlow 使用 `WEFLOW_API_TOKEN` 或 `WEFLOW_HTTP_TOKEN`。
4. 启动控制台：
   - 普通启动：`start_sidebar.cmd`
   - 强制启动 WeFlow 并自动安装依赖：`start_sidebar.cmd --weflow on --install-weflow-deps always`
   - 只启 HTTP 服务：`start_sidebar.cmd --mode server --port 8765`
5. 新安装默认配置为 `send_driver=bridge_outbox`、`send_backend=wechat_native_http`、`send_enabled=false`。这表示控制台默认指向当前真实桥链路，但不会自动发送；只有在 Native 探测、模型 key、发送 worker 和审核链路都确认后，才手动开启发送。

## 3. Native hook 迁移

控制台“诊断/Native 迁移探测”是首选入口。它会读取微信版本、native HTTP 状态、消息目录候选和 hook 残留清单，并写入 `data/native_diagnostics/native-migration-latest.json`。

命令行流程：

1. 关闭微信。
2. 如需构建当前 hook artifact：
   `powershell -ExecutionPolicy Bypass -File .\scripts\build_wechat_native_hook.ps1`
3. 部署 hook：
   `powershell -ExecutionPolicy Bypass -File .\scripts\deploy_wechat_native_hook.ps1 -WeixinRoot "C:\Path\To\Weixin"`
4. 重新打开微信，确认 native HTTP：
   `python -m app.personal_wechat_bot.main --data-dir data send-driver-probe`
5. 跑迁移探测：
   `python -m app.personal_wechat_bot.main --data-dir data native-migration-probe --force-scan`
   如只想预览不写入诊断文件，可追加 `--no-persist`；小机器上可用 `--max-depth`、`--max-entries`、`--limit` 控制扫描范围。
6. 在控制台点击 Native 迁移探测或查看 `data/native_diagnostics/native-migration-latest.json`，确认状态为 `ready` 后再开启真实发送。

清理边界：

1. 必须保留 `vendor/artifacts/wechat-native-411053/version.dll`，这是当前部署基线。
2. 保留 `vendor/reference/WeChat-Hook-aixed` 的源码、工程文件和必要运行库，便于在新机器重建；可清理其中 `.git`、`x64`、`x64_Version` 等历史/构建输出。
3. `vendor/reference/wechat-binaries`、`vendor/reference/WeChatFerry-gitee`、`vendor/reference/wxbot`、`vendor/artifacts/wechat-hook-411053-image-experimental` 属于历史参考或实验产物，迁移探测会列入 dry-run 清单，不会自动删除。
4. `vendor/reference/WeFlow-gitcode/node_modules`、`.git`、`dist-electron` 属于可重建缓存/构建输出；迁移到新机器后通过控制台“安装依赖”或 npm 重新生成。

不要恢复 WCF/wcferry、剪贴板、焦点窗口或前台键鼠发送链路。当前正式发送基线是 `bridge_outbox -> send_bridge_worker -> wechat_native_http`。

## 4. 可迁移与应重建的数据

可迁移：

1. `data/config.json`
2. `data/runtime_cards*`，包含全局与通道级人物卡/技能/任务卡
3. `data/api_key_models.local.json`、`data/api_keys.local.md`，仅在确认不会泄露密钥时迁移
4. `data/send_bridge/outbox.jsonl`
5. `data/send_bridge/acks.jsonl`
6. `data/send_bridge/synced_acks.json`
7. `data/send_bridge/accepted_reverify.json`
8. `data/send_bridge/.bridge_worker.lock` 仅作为证据保留，不建议跨机器复用运行锁

建议重建：

1. `data/conversation_ledgers` 和 `data/conversation_sessions`，正式启动前可清空重新生成。
2. `data/confirm_queue.sqlite` 和 `data/confirm_queue.jsonl`，审核队列属于运行历史。
3. `data/send_audit.sqlite` 和 `data/send_audit.jsonl`，发送审计属于运行历史；真实桥证据另存于 `data/send_bridge`。
4. `data/sidebar_state.sqlite` 和 `data/weflow_sidebar_state.json`，控制台投影可重建。
5. `data/native_diagnostics`，每台机器重新探测。

迁移前建议先执行：

1. 预览清理项：`python -m app.personal_wechat_bot.main --data-dir data cleanup-artifacts`
2. 确认候选只包含可再生缓存/诊断后再应用：`python -m app.personal_wechat_bot.main --data-dir data cleanup-artifacts --apply`

该命令会清理浏览器 profile 缓存和 native hook 历史诊断，保留配置、对话关键状态、send_bridge 证据链、file_workspace 以及最新 native migration 探测文件。

## 5. 清理历史边界

控制台“清空历史”会清理对话、会话、文件中间层、任务、审核队列、发送审计、Agent cursor 和 WeFlow 运行痕迹。

必须保留：

1. 配置文件和模型/key 配置。
2. 技能/人物卡/任务卡。
3. `data/send_bridge` 中的 outbox、acks、synced marker、accepted reverify marker 和 worker lock 证据。

清理前应停止 Agent/WeFlow 写历史的 worker；控制台会在检测到活跃历史写入者时阻断直接清理。

## 6. 状态库说明

当前存储策略：

1. 对话 ledger：Markdown 真源，必须严格映射真实线性对话。
2. 确认队列：`confirm_queue.sqlite` 是权威库，`confirm_queue.jsonl` 是兼容投影。
3. 发送审计：`send_audit.jsonl` 是证据流，`send_audit.sqlite` 是读索引。
4. 控制台状态：`sidebar_state.sqlite` 是权威库，`weflow_sidebar_state.json` 是兼容投影。
5. send bridge：JSONL 证据链，历史清理必须保留。

## 7. 验收命令

1. `python -m app.personal_wechat_bot.main --data-dir data preflight`
2. `python -m app.personal_wechat_bot.main --data-dir data send-driver-probe`
3. `python -m unittest tests.test_send_commands tests.test_send_bridge_worker tests.test_sidebar_api -v`
4. `node --check app\personal_wechat_bot\ui\sidebar\app.js`

正式开启真实发送前，控制台应显示：

1. send backend 为 `wechat_native_http`
2. native HTTP 可达，微信登录状态可读
3. send bridge worker 配置 `matched`
4. 待审、入桥、未验没有历史残留
5. Native 迁移探测状态为 `ready` 或无阻断项
