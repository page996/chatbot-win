# 后续完整计划书

## 0. 当前基线

本文用于承接后续所有改进任务，和 `计划书.md` 的流水记录分工如下：

- `计划书.md`：记录每一轮具体完成内容、验证命令、真实数据状态和下一步残留。
- `后续完整计划书.md`：作为后续长任务总纲，覆盖所有待做方向、优先级、验收标准和测试矩阵。

当前已确认的基线：

1. 本地 native hook 是唯一正式非前台输出端口。
2. WCF/wcferry 链路已弃用，不再恢复。
3. 不走前台模拟输入、剪贴板、焦点窗口或键鼠自动化发送。
4. 文本真实发送已跑通。
5. 普通文件真实发送已跑通。
6. 图片/GIF/表情发送暂不做，保持 unsupported，除非后续明确要提升活人感。
7. `bridge_outbox -> send_bridge_worker -> wechat_native_http -> ack/readback -> confirm queue/ledger/task` 是当前正式发送链路。
8. 当前安全运行态（2026-07-10 收尾阶段）：
   - `send_driver=bridge_outbox`
   - `send_backend=wechat_native_http`
   - `send_enabled=false`
   - 旧 send bridge worker PID `15232` 已停止，锁文件仅作为证据保留
   - 本轮不做真实微信发送 probe，不向真实 outbox/acks 写测试记录
9. 当前总台和审核状态已收敛：
   - task active=0
   - confirm pending=0
   - confirm approved=0
10. 已跑通证据必须保留：
    - `data/send_bridge/outbox.jsonl`
    - `data/send_bridge/acks.jsonl`
    - `data/send_bridge/synced_acks.json`
    - `data/send_bridge/accepted_reverify.json`
    - `data/send_bridge/.bridge_worker.lock`
11. 通道、ledger、session 已切为三个 SQLite 权威库；JSON/JSONL/Markdown 只作为可读投影。
12. 部署采用“清空旧历史、空库启动”的口径，不迁移既有通道或对话文件；真实结构和投影恢复在隔离临时目录验证。

## 1. 总目标

最终目标是把当前系统从“链路已打通”推进到“可长期运行”的状态：

1. 微信 4.x 当前版本下，文本和普通文件可通过本地 native hook 稳定非前台投递。
2. 上游输入、通道状态、对话 Agent、任务总台、审核队列、发送桥、ledger、memory 和前端展示完全对齐。
3. 多通道、多用户消息、账号本人手动消息、agent 自发消息互相隔离，不串话、不误触发、不重复回复。
4. 长对话中优先基于当前聊天底部语境接话，而不是追答旧历史。
5. 文件消息在输入、输出、ledger、通道文件、memory summary、entities 中形成闭环。
6. 历史清理和实机测试不会破坏已经跑通的 native 发送证据。
7. 所有关键状态都有可解释、可测试、可恢复的生命周期。

## 2. 不变量

后续任何改动都不能破坏以下不变量：

1. 无 talker 或不在白名单的 talker 不允许真实发送。
2. 自动模式不应进入人工审核门禁；手动模式必须保留审核。
3. `send_bridge_worker` 必须单实例运行，不允许双 worker 同时消费同一 outbox。
4. 任何未知发送结果不能立即重发，必须 quarantine 或保留 accepted/unverified，防止重复真实投递。
5. 对已 terminal 的 bridge ack，后续失败 ack 不能降级已经 sent 的 queue/ledger/task。
6. `accepted` 只表示端口已接收或原生调用已返回，不能伪装成 `sent/verified`。
7. 文件发送在读回唯一匹配前只能是 accepted/unverified。
8. 历史清理不能删除 `data/send_bridge` 的 outbox、ack、synced marker、accepted reverify marker 和 worker lock。
9. UI 探针、sidebar 自测项、诊断项不能长期污染任务总台。
10. 通道 scope、Agent cursor、WeFlow 拉取范围必须一致，选 A 通道不能消费 B 通道事件。
11. 账号本人手动发送必须和 agent 自发消息、用户消息区分。
12. 群聊内 sender 身份必须参与聚合，不能把账号本人当群友回复。
13. 任务总台 SQLite 是任务权威库；JSON 只是投影。
14. 任何修复都要有测试或可复现验证命令。
15. 未登记、未知、非好友通道必须 fail closed；发送 receiver 必须与通道库中的 wxid/roomid 精确一致。
16. worker 每次 wire retry 前必须重新检查发送总开关、driver/backend、receiver 和通道授权。
17. `accepted_unverified` 可能已经投递，不允许自动或手工重投，只能等待回读确认。
18. `conversation_channels.sqlite`、`conversation_ledger.sqlite`、`conversation_sessions.sqlite` 是权威库；缺失投影只能从数据库重建。

## 3. 优先级分层

### P0 运行安全

必须先做，直接影响真实投递和长期运行：

1. 发送桥单实例、锁心跳、stale config 修复。
2. 总台任务 active/running/queued/waiting 收敛。
3. 审核队列和 bridge ack 同步一致。
4. 无 talker 阻断和通道白名单。
5. 历史清理边界。
6. 旧测试项、旧 worker 项、旧 backend blocker 的自动退休。

### P1 对话正确性

影响 agent 是否像正常人一样接话：

1. 多通道独立聚合。
2. 私聊和群聊身份隔断。
3. 账号本人手动消息识别。
4. 底部语境优先。
5. 主题提取和消息聚合质量。
6. 重复回复惩罚。
7. 主动打招呼和主动接话策略。

### P2 文件和记忆闭环

影响长期记忆、文件理解和可追溯性：

1. incoming 文件入库和解析。
2. agent outgoing 文件记录。
3. file-only 回复进入 ledger 和 memory。
4. memory summary/entities 合并确定性文件字段。
5. 文件读回验证和歧义处理。

### P3 前端和运维体验

影响测试效率和故障定位：

1. 总台/WeFlow/Agent/发送桥状态三方对齐。
2. 审核栏状态切换延迟解释和压缩。
3. 明确的重投、移除、保留按钮。
4. 诊断面板和一键导出。
5. 文案和参数语义统一。

### P4 扩展能力

当前不阻塞主线：

1. 图片/GIF/表情包发送。
2. 更强的 native hook 自检和版本迁移。
3. 多账号或多微信实例。
4. 更复杂的长期记忆检索。

## 4. 阶段计划

### 阶段 A：运行态收敛加固

目标：让长跑后总台、审核、bridge、ledger 不出现假 active、假 pending 或僵死项。

任务：

1. 总台任务状态收敛
   - 检查 `running` 任务是否有对应活 worker。
   - worker 不存在时按任务类型终态化。
   - bridge ack 已 terminal 时，send task 必须 completed/failed。
   - reply 已写入 ledger 时，reply task 必须 completed。
   - UI 探针和诊断任务不进持久总台，已进库时自动净化。

2. 发送审核状态收敛
   - approved 项必须有明确下一步：等待手动发送、自动发送中、已入桥、已发送、失败、已退休。
   - auto 模式下旧 sidebar 自测项自动退休。
   - 普通 approved 回复不能被误退休。
   - failed 但后续收到 sent ack 时允许修复。
   - sent 不能被后续 failed 降级。

3. bridge worker 收敛
   - stale config 检测必须比较完整 backend signature。
   - 配置变化时旧 worker 要停止或被新 worker 接管。
   - lock heartbeat 不能在长发送或大 backlog 时过期。
   - worker 崩溃后 supervisor 有重启上限和可见状态。
   - 无法确认结果时 quarantine，不重复真实发送。

4. 真实数据巡检
   - 每轮结束读取：
     - task counts
     - confirm counts
     - bridge pending
     - worker config
     - native health
   - 发现 active 但 bridge pending=0 时，必须解释来源。

验收：

- 连续刷新 sidebar 不产生新残留任务。
- `active=0` 时 UI、SQLite、JSON 投影一致。
- `pending_count=0` 时没有 queued_to_bridge 假 pending。
- 相关测试：`tests.test_task_manager`、`tests.test_send_commands`、`tests.test_send_bridge_worker`、`tests.test_sidebar_api`。

### 阶段 B：历史清理和证据保留

目标：让实机测试前后可以安全清历史，不误伤 native 发送证据。

任务：

1. 清理边界明文化
   - 可清：
     - backend events
     - conversation sessions
     - conversation ledgers
     - channel state
     - processed messages
     - task manager
     - send audit
     - agent runtime cursor
   - 必须保留：
     - send bridge outbox/ack/synced/reverify/lock
     - native hook artifact 和备份记录
     - config
     - launch/runtime audit 基础配置
   - 三个对话权威库、WAL/SHM 和可读投影必须同批清理，禁止只删数据库或只删投影。

2. 运行中清理保护
   - Agent worker 运行中阻断直接清理。
   - WeFlow worker 运行中阻断直接清理。
   - pull-once/backfill 运行中阻断直接清理。
   - 前端 helper 可先停进程再清理。

3. 清理报告
   - 输出 deleted/skipped/failed/preserved。
   - 对锁文件和正在占用文件说明原因。
   - 清理后自动刷新 task/queue/bridge 探针。

4. 回归测试
   - 清理不会删除 `data/send_bridge`。
   - 清理会删除 `runtime/agent_state.json`。
   - 锁文件失败时 partial_error 可见。
   - shutdown helper 不重复启动。
   - 清理后重开三个 store，旧通道、旧消息、旧 session/reset 事件均不得复活。
   - 新建的三个 schema v1 空库必须通过 integrity、表、列、索引和行数合同检查。

验收：

- 清历史后 native send probe 仍 ready。
- 清历史后 bridge outbox/ack 仍存在。
- 清历史后 agent cursor 已清空。
- 清历史后无旧 UI 探针任务残留。

### 阶段 C：通道隔断和身份识别

目标：把每个私聊/群聊视为独立通道，并正确识别用户、账号本人、agent。

任务：

1. 通道库权威化
   - `conversation_channels.sqlite` 是权威库，`channel.json/index.json` 是可读投影。
   - 只对在库且已授权通道负责。
   - 旧窗口 OCR/标题 fallback 不再作为主要通道来源。
   - wxid/roomid 是发送权威地址。
   - display name、emoji、符号名只做展示和别名。
   - outbox receiver 必须与持久化 wxid/roomid 精确一致；未知、非好友和未登记项在 wire 前阻断。

2. talker scope 对齐
   - 前端选中的 talkers 必须映射到同一组 conversation_id。
   - Agent state 展示的“逐通道聚合：N 通道”必须来自同一 scope。
   - WeFlow 拉取、Agent tick、send bridge 白名单必须共享同一组 talker。

3. 身份边界
   - `user`：对方或群成员。
   - `owner`：当前账号本人手动发送。
   - `agent`：系统生成并通过 bridge 发送。
   - 群聊中必须记录 sender wxid/name。
   - owner 消息不能被当成新的群友消息触发回复。
   - agent 消息不能被当成用户消息重复回复。

4. display name 和符号兼容
   - emoji、符号、乱码样式标题不能被概览过滤。
   - channel state manager 和概览展示规则一致。
   - 对不可信 legacy 标题只隐藏未登记项，不隐藏已登记通道。

5. 游标隔离
   - 每个 scope 有独立 Agent cursor。
   - 选 A 不消费 B。
   - 切回 B 后仍能处理 B 未读。
   - 全局 tick 不破坏 scoped cursor。

验收：

- 多私聊并行时每个通道只回复自己的消息。
- 群聊中 owner 发言不触发 agent 对 owner 的回复。
- agent 自己刚发的消息不触发重复回复。
- emoji/符号名通道在概览和状态管理器都可见。

### 阶段 D：完整对话 Agent 闭环

目标：Agent 启动后完成一次完整聊天，并能持续工作。

标准流程：

1. 启动 Agent。
2. 读取当前 session 的对话文件。
3. 按通道做主题提取。
4. 聚合当前底部语境和未处理消息。
5. 生成任务分发。
6. 管道分配资源并执行。
7. 结果回推到对话 Agent。
8. Agent 生成接话。
9. 进入发送链路。
10. bridge ack 回写审核、ledger、task、memory。
11. 下一轮 tick 继续处理新消息。

任务：

1. session 读取
   - Agent tick 前读取 current session id。
   - 加载 session 内最近 N 条消息。
   - 区分当前 session 和历史 session。
   - 对过长历史使用摘要，不追答旧消息。

2. 主题提取
   - 每通道独立主题。
   - 同一通道多个近期用户消息可聚合成一个主题。
   - 无关主题可拆成多个回复，但必须符合当前聊天底部可见语境。
   - 旧主题已回复后默认关闭，除非用户再次提起。

3. 消息聚合
   - 私聊：按时间窗口、未回复用户消息、最后 assistant/owner 消息之后的增量聚合。
   - 群聊：按 sender、mention、上下文相关度聚合。
   - 文件消息和文本消息合并成一个上下文包。
   - 低价值系统消息、重复回执、agent 自己消息排除。

4. 回复决策
   - 有新用户消息才主动接话。
   - 新好友/新私聊可生成一次打招呼。
   - 无新增消息时不重复回复。
   - 对已失败但未可见的回复，要判断是否需要重试，不盲目生成新文本。

5. 发送策略
   - auto 模式直接入桥。
   - manual 模式进入审核。
   - 无 talker 阻断。
   - 非白名单阻断。
   - 失败原因必须写入 audit。

验收：

- 连续 tick 无新消息不重复回复。
- 两个无关主题在同一通道内可生成合理分段回复。
- 多通道新消息可逐通道回复。
- 新私聊可主动打一次招呼。
- 失败发送不会造成无限生成新回复。

### 阶段 E：长对话聚合质量

目标：让 agent 更像在当前聊天窗口底部自然接话。

任务：

1. 底部语境优先
   - 最近可见消息权重最高。
   - 历史摘要只作为背景，不作为逐条待回复项。
   - 用户已经转话题后不追答旧话题。

2. 重复惩罚
   - 对近 N 条 assistant/agent 回复做相似度检查。
   - 对相同主题重复回复降权。
   - 对已关闭主题不再主动开启。
   - 允许用户要求“再说说”“继续”“展开”时续答。

3. 主题生命周期
   - `open`：有新用户消息待处理。
   - `responded`：已生成回复，等待发送结果。
   - `sent`：发送成功或 accepted。
   - `closed`：无后续追问。
   - `reopened`：用户再次提起。

4. 主动行为
   - greeting 和 pending reply 可以并行，但每通道需要节流。
   - 主动打招呼只对新通道或长期无 agent 消息通道生效。
   - 主动接话必须有明确上下文触发。

5. prompt 质量
   - 输入中明确当前通道、身份、最后几条消息。
   - 提醒模型不要逐条追答旧历史。
   - 文件不可读时诚实说明，不虚构内容。
   - 回复短、自然、少解释系统行为。

验收：

- 用户切换话题后 agent 不再回复旧话题。
- 多轮 tick 回复不会高度重复。
- 文件不可读时回复不伪造文件内容。
- 群聊中不对每个群友逐条机械回复。

### 阶段 F：文件输入、输出、ledger 和 memory 闭环

目标：文件消息在系统内有稳定、可追溯、可记忆的结构。

任务：

1. incoming 文件
   - WeFlow/后端事件捕获文件。
   - 写入 file workspace。
   - 生成 attachment summary。
   - 按类型解析：
     - text/csv/json/md
     - doc/docx
     - ppt/pptx
     - xls/xlsx
     - pdf
   - 解析失败时保留失败原因，不丢文件实体。
   - 原件按 SHA-256 存入 `_blobs/<prefix>/<digest>`，staged 原件优先 hardlink，失败时回退 copy。
   - 解析前校验摘要；blob/staged 原件保持只读，cleanup 只回收无 manifest 引用 blob。

2. outgoing agent 文件
   - agent 生成文件后必须有 file artifact id。
   - 发送前写入 ledger draft。
   - 发送后写入 send status、bridge_id、external_message_id。
   - accepted/unverified 和 sent/verified 区分。

3. ledger 适配
   - file-only 回复不能因为无文本而丢失。
   - 消息 entry 要包含：
     - origin
     - direction
     - role
     - file_id
     - local_path
     - display_name
     - mime/type
     - size
     - send_status
     - bridge_id
     - external_message_id

4. memory 适配
   - `summary.md` 保留 Recent Files。
   - `entities.json` 合并确定性文件字段。
   - LLM 摘要不能覆盖或吞掉真实 bridge/file 字段。
   - 重要文件可形成可检索实体。

5. 文件发送验证
   - 文本文件和普通文件通过 `/send_file_msg`。
   - 文件 readback 唯一匹配才升级 sent。
   - 同名同大小多条 outgoing 文件保持 accepted/unverified。
   - 文件不存在、路径不允许、超大小应明确 failed。

验收：

- 用户上传文件后，ledger、file workspace、memory 都能看到同一文件实体。
- agent 发送文件后，ledger、confirm queue、bridge ack、memory 都有一致字段。
- file-only agent 回复可被记忆摘要捕获。
- 并发同名文件不会误验 sent。

### 阶段 G：前端体验和操作面板

目标：让实机测试可控、可解释、可恢复。

任务：

1. 总台
   - active/running/queued/waiting/failed 数字和 SQLite 权威状态一致。
   - worker 任务、bridge 任务、probe 任务区分显示。
   - 可查看每个任务的来源、scope、external_id、bridge_id。
   - 提供旧项处理按钮：
     - retry
     - remove
     - retire obsolete
     - inspect audit

2. WeFlow 栏
   - talker 数、通道数、拉取数、上下文回填开关语义清楚。
   - “只回填上下文”按钮不应长期激活且无法关闭。
   - 参数展示和真实后端配置三方对齐。

3. 对话 Agent 栏
   - 显示逐通道聚合，而不是误解为跨通道混合。
   - 显示当前 scope、talkers、pending user count、last tick。
   - 提供启动一次、连续启动、停止、清 cursor。
   - 显示 greeting、proactive、reply 的分类计数。

4. 发送审核栏
   - pending/approved/queued_to_bridge/accepted/sent/failed 清晰分栏。

   - 支持通道级暂停、恢复、删除、隐藏。

6. 测试按钮
   - 通道内生成一条测试回复。
   - 通道内投递一个上传文件。
   - 两者都必须经过当前发送控制逻辑。
   - auto/manual 行为和真实回复一致。

验收：

## 7. 推荐部署验收顺序

worker 终态化、投递 fail-closed、三个对话权威库、附件内容寻址和 sidebar 数据库合同展示均已完成自动化与本地 UI 验收。下一入口是用户清空旧历史后的新链路实测，而不是迁移旧通道/对话文件。

推荐顺序：

1. 保持 `send_enabled=false`，停止 Agent、WeFlow 和 send bridge worker。
2. 使用控制台“清空历史”，确认三库、WAL/SHM 和可读投影被同批删除，同时 `data/send_bridge` 证据完整保留。
3. 重新启动 sidebar，打开“历史/存储边界”，确认三个 schema v1 合同均为 `integrity_check=ok` 且业务表为空。
4. 让一个已确认好友私聊和一个已登记群聊各产生新消息，确认通道、ledger、session 数据与六类可读投影从空状态首次生成。
5. 对未知私聊、非好友、未登记群聊和 receiver 不匹配项验证 fail closed；这一步仍不打开真实发送。
6. 只有在通道身份、receiver、审核和 worker 配置均确认后，才由用户决定是否开启真实发送并做单条文本/文件验收。

## 8. 用户实机测试清单
