# 维护改良计划

状态：有效

更新时间：2026-07-11

## 使用边界

1. 当前架构事实以代码、SQLite schema、测试和 `打造计划.md` 为准；本文只记录尚未完成且仍有维护价值的改良方向。
2. 部署、迁移、历史清理和 `send_bridge` 保留边界以 `迁移部署方案.md` 为准。
3. 完成的条目从本文移除，实施过程与验收证据进入 Git 提交，不在仓库累积阶段流水账。
4. 任何改良都不得恢复文件投影回导、页面 OCR 对话输入、前台窗口绑定、剪贴板发送或默认开启真实发送。

## P0 持续发布门槛

1. 对话、存储、任务或前端状态机变更必须通过全量 Python 测试、Python 编译、JavaScript 语法检查和 `git diff --check`。
2. 共享行为变更必须运行 `audit-plan`，保持 `plan_residual_count=0`；存储变更必须用 `storage-status --no-sizes` 核对 schema、索引、完整性和行数。
3. 前端交互变更必须覆盖桌面与窄屏，验证同作用域串行、刷新合并、保存不覆盖编辑、关闭面板不被迟到响应重开、历史清理冻结和错误恢复。
4. 发布前确认 `send_enabled=false`；涉及历史清理时，逐项复核 `data/send_bridge` 证据文件哈希。

## P1 可靠性改良

1. 将当前人工 Playwright 验收固化为可重复的浏览器测试，覆盖桌面/窄屏、请求竞态、非法 JSON、超时、重复点击和历史清理互斥。
2. 对 WeFlow 多 talker 后台链路做长时间 soak：验证游标单调、跨会话并发、同会话串行、去重稳定、`stalled`/慢 tick 指标和 worker 重启收敛。
3. 持续审计长耗时操作是否都具备 `TaskRecord`、取消/恢复事件、资源门控和跨进程 claim 语义；不得只在前端伪造任务状态。
4. 扩大故障注入覆盖：SQLite busy/WAL、进程锁陈旧、bridge ack 延迟/乱序、worker 中断和配置热更新，确保失败后不会重复发送或重复回复。

## P2 存储与检索

1. 如需语义检索，优先增加可重建的 SQLite FTS5 派生索引；ledger 和文件正文仍以现有 SQLite/内容寻址存储为权威，索引不得反向写回事实数据。
2. schema 升级必须有显式版本、前向迁移、回滚前备份和旧版本拒绝策略；禁止从 JSON、JSONL 或 Markdown 投影恢复权威库。
3. 为确认队列、发送审计和任务事件定义可验证的导出/归档策略；`send_bridge` 只允许人工、带哈希的证据归档，不纳入自动清理或自动重放。

## P3 能力扩展

1. 图片/GIF 真实发送继续保持 unsupported，直到 native 接口、WeFlow/数据库回读、重复发送防护和 confirm 灰度全部具备可验证证据。
2. 新微信版本先更新 native compatibility matrix 并通过只读迁移探测；未经验证不得沿用 `4.1.10.53` 的偏移、DLL 或发送能力声明。
3. 新媒体或文件类型必须先进入隔离 workspace、大小/类型校验和结构化 manifest，再决定是否解析；不得直接扩大可执行或解压边界。

## 完成标准

每个改良项必须同时交付实现、风险边界、自动化回归、运行态验收和文档更新。只有在默认安全配置不变、无兼容回导、无历史计划残留且 Git 工作树干净时，才可合并到 `main`。
