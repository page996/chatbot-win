## 背景结论

此前总台设计已经完成第一版落地：`TaskStatusStore` / `SchedulerStore` 管任务，`ResourceScheduler` 管资源预算，sidebar 已经拆出总台资源池、总台进程、通道状态管理器、操作历史和 WeFlow 操作历史。

现在的主要问题不是“没有总台”，而是通道状态管理器还偏投影层：

- `ChannelStateStore` 会保存 sidebar 投影，但刷新时会从 channel registry、task manager、ledger 重新生成并 `replace_all`。
- 这意味着未来人工暂停、优先级、等待原因、备注、置顶等控制状态如果直接写进 `channel_state.sqlite`，会被下一次投影刷新覆盖。
- 前端 Channel Lane 已经能展示文件、回复、任务和资源，但还没有稳定的后端控制层供 UI 操作。
- 旧计划文档把“已完成设计”和“后续迁移项”混在一起，容易把已落地的总台当成未完成任务。

## 目标边界

通道状态管理器只负责单个会话的运行状态，不直接执行任务：

- 当前主题、主题队列、历史主题。
- 会话内任务：回复、文件解析、AI 分析、发送、等待/阻塞。
- 文件状态：解析、chunk、OCR/ASR、AI summary/key points。
- 回复状态：drafted、待审、入桥、已发、失败。
- 人工控制：暂停、恢复、优先级、等待原因、备注、置顶/静音。
- 资源摘要：该通道正在使用/排队的资源池。

总台继续只负责跨会话资源和调度：

- GPU / LLM / media_cpu / file_io / wechat_io / send_bridge。
- `claim_next()` 的资源预算、并发限制、依赖检查、concurrency key mutex。
- WeFlow tick、诊断、设置、审计仍不进入 Channel Lane。

## 当前推进顺序

进度快照：

- Phase 1 持久控制层：已完成。
- Phase 2 后端控制 API：已完成。
- Phase 3 调度接入：已完成，`claim_next()` / dispatch preview 已识别 `paused`、未到期 `snoozed` 和通道优先级 overlay，前端总台会展示调度预览。
- Phase 4 前端操作：已完成第一版轻量控制条。
- Phase 5 任务事件详情：已完成第一版前端入口和内联事件列表。

### Phase 1：持久控制层（已完成）

后端给 `ChannelStateStore` 增加 `control` overlay：

- `mode`: `active | paused | muted | snoozed`
- `priority`: 0-100
- `pinned`: true/false
- `wait_reason`
- `operator_note`
- `snoozed_until`
- `updated_by`
- `updated_at`

sidebar 刷新时，实时投影仍从 task/ledger/channel 生成，但必须把旧的 `control` 合并回新投影。验收点：

- 人工暂停不会被 `build_sidebar_state()` 覆盖。
- `effective_status` 能表达 `paused/snoozed/muted` 与任务态的叠加结果。
- 不存在真实 channel 时，状态读取仍不凭空创建空目录。

### Phase 2：后端控制 API（已完成）

新增或扩展 API：

- `POST /api/channel-state`：pause/resume/mute/snooze/set-priority/note。
- 返回更新后的 `channel_state` 和 `task_manager`。
- 控制状态只影响调度/展示，不直接删除任务。

### Phase 3：调度接入（已完成）

让 `TaskStatusStore.claim_next()` 或其调用方识别通道控制：

- `paused` 通道不领取新任务。
- `snoozed` 到期前不领取新任务。
- `priority` 参与会话级排序。
- 等待用户确认的 send task 继续保持 `waiting`，不占资源槽。
- 过期 `snoozed` 不再覆盖 `effective_status`，展示状态与调度领取语义一致。
- 总台调度策略面板展示 dispatch preview：可领取数、阻塞数，以及阻塞原因。

### Phase 4：前端操作（第一版已完成）

Channel Lane 展开区增加轻量操作：

- 暂停/恢复。
- 优先级调整。
- 等待原因/备注。
- 事件详情入口。

UI 原则：只展示必要操作，不把 Channel Lane 做成另一个混杂任务台。

已完成第一版：

- Channel Lane 摘要显示 `effective_status`、控制模式和优先级。
- 展开区增加“通道控制”轻量面板。
- 支持暂停/恢复、静音、稍后处理、置顶、优先级、等待原因、备注。
- 前端操作统一调用 `POST /api/channel-state`。
- 操作 scope 使用 `channels:control:*`，不会进入 Channel Lane 任务列表。

### Phase 5：任务事件详情（第一版已完成）

从 `/api/tasks` event stream 读取任务事件：

- 展开单个任务可看 claimed/transition/update/finish_external。
- WeFlow 普通 tick 不进入主任务事件；失败/异常可升格。

已完成第一版：

- 总台任务和 Channel Lane 任务卡片都增加“事件”入口。
- 前端调用 `/api/tasks` 的 `events` action 读取事件流。
- 内联展示 `created / updated / claimed / transition:* / finish_external` 等关键事件。
- 事件列表只展示摘要字段，不把原始 payload JSON 整块塞进界面。
- HTTP 路由测试覆盖 `/api/tasks` events 返回。

## 本轮验收

已完成的 Phase 1/2 验收：

- 新增 `ChannelStateStore.patch_control()`。
- 新增投影合并函数，保留 `control`。
- `_channel_state()` 写入投影前合并旧控制状态。
- 增加单元测试和 sidebar 投影测试。
- 跑 `test_channel_state_store`、`test_sidebar_api`、相关 py_compile。

已追加验收：

- `POST /api/channel-state` 已接入 sidebar server，并补 HTTP 路由测试。
- `TaskStatusStore.claim_next()` 已识别通道控制：
  - `paused` 通道不领取新任务。
  - `snoozed` 且 `snoozed_until` 未到期时不领取新任务。
  - `snoozed_until` 已过期时不继续阻塞领取。
  - `priority` 作为会话级 overlay 参与 dispatch score。
- dispatch preview 会标出 `channel_paused:*` / `channel_snoozed:*`，便于前端解释“为什么没被领取”。
- 前端已把 dispatch preview 翻译成人可读原因，例如通道暂停、稍后处理、资源忙、依赖未完成、同作用域串行。

本轮复核修复：

- `POST /api/channel-state` 现在同时返回 `task_manager`，与计划中的 API 契约一致。
- `_channel_state()` 在没有真实 channel registry、但已有 `channel_state.sqlite` 时不再 `replace_all([])` 擦掉 control overlay；它会只读返回已有 `channel_states`。
- 过期 `snoozed` 不再让通道显示为“稍后”，避免 UI 状态和实际调度策略错位。
- dispatch preview 现在会按真实调度顺序模拟资源、通道和 concurrency key 槽位占用，避免多个 queued task 竞争同一槽位时都被错误展示为“可领取”。
- 补齐计划中提到但首轮漏掉的“置顶”控制：`control.pinned` 会持久化、参与通道/调度排序、支持 `pin/unpin` API，并在前端轻量面板中显示和保存；dispatch preview 会暴露 `channel_pinned` 方便解释排序。
- 增加 sidebar HTTP 动态 roundtrip 验收：创建通道和任务后，通过 `/api/channel-state` 置顶/暂停，再重新读取 `/api/state`，确认 Channel Lane 控制态和 dispatch preview 阻塞原因同步更新。
- 增加本机 Microsoft Edge headless 冒烟验收：在临时 sidebar server 上加载真实页面并执行前端 JS，确认 Channel Lane、置顶复选框和 dispatch preview 实际渲染到 DOM；缺少 Edge 的环境会跳过该用例。

本轮可选项与扫描结论：

- 抽出 `conversation/channel_control.py`，统一 `mode`、布尔值和 `snoozed_until` 是否仍有效的解析逻辑，避免 `ChannelStateStore` 与 `TaskStatusStore` 各自维护一份判断。
- 新增 `test_channel_control`，固定 `parse_bool`、`normalize_control_mode`、`snooze_is_active` 的边界语义；其中无效或空的稍后时间继续按“保守阻塞”处理。
- 修正前端 `shortTime()` 的 ISO 时间显示：`snoozed_until` 等 UTC 时间会按浏览器本地时间展示，避免“稍后到”保存后摘要显示成 UTC 小时。
- 复扫新增链路：`/api/channel-state`、`ChannelStateStore`、`TaskStatusStore.claim_next()`、dispatch preview、前端 Channel Lane 控制面板和 HTTP/Edge 验收测试已形成闭环；暂未发现需要立即修复的逻辑断点。

## 实机后新增残留：输出闭环与上下文适配

2026-07-08 实机测试后复核：

- WeFlow 拉取、历史回填、多个 background tick 暂未发现通道状态异常；`backend_events.jsonl` / `hook_events.jsonl` 各 23 条，`channel_state.sqlite` 有 1 个通道，`file_index.sqlite` 有 10 个文件索引。
- 当前 ledger 只有 `user=22`、`self=1`、`assistant=0`，`send_bridge/outbox.jsonl` 与 `acks.jsonl` 都为空；说明已验证“输入/入账/附件解析/通道展示”，但还没有验证 agent 真实生成回复、入账、排队、发送和回执。
- `send_enabled=true`、`mode=auto`、`send_confirm_required=false`，但 sidebar readiness 显示 `bridge_outbox_worker_down`，并提示 `send_backend=dry_run`；在启动真实 worker 前，应先回到 confirm/可审计模式完成输出闭环验收，避免自动实发。

后续优先级：

1. 文本输出闭环：用受控单会话触发 `MessageProcessor -> generate_reply -> append_reply -> reply_gate -> send task/outbox`，先要求 confirm 或 dry-run 可审计，再考虑真实桥发送。
2. 多用户消息与主题提取：把连续多条用户消息聚合成一个可回复主题，避免每条消息各自生成碎片任务；主题要写入 `topic_id/topic_title`，并反映到 Channel Lane。
3. agent 产出文件：为 `ReplyCandidate.attachments` / tool output 建立出站文件验收，确保 agent 自己生成的文件进入 file workspace、ledger、channel state、send bridge contract。
4. 对话文件与记忆适配：memory maintainer 需要识别 assistant 回复、assistant 附件、用户多消息主题和文件摘要，避免记忆只基于入站用户消息。
5. 实机验收补强：补一组“真实文本/文件输出”测试场景，覆盖 assistant entry、send task、outbox/ack、Channel Lane reply_state、文件状态与 memory 更新。
