# 成熟项目参考对齐报告

更新时间：2026-06-30

## 参考项目与文档

本轮架构对齐只采用成熟项目/官方文档作为参照，不把它们直接引入为运行时依赖。

1. LangGraph/LangChain Memory
   - 官方文档：<https://docs.langchain.com/oss/python/concepts/memory>
   - 参考点：短期记忆按 thread/session 隔离；长期记忆按 namespace/key 存储，可跨会话检索或注入。
2. OpenAI Agents SDK Sessions
   - 官方文档：<https://openai.github.io/openai-agents-python/sessions/>
   - 参考点：Session 负责跨多轮 agent run 维护对话历史，应用侧仍拥有状态、审批和工具编排边界。
3. LlamaIndex Memory
   - 官方文档：<https://developers.llamaindex.ai/python/framework/module_guides/deploying/agents/memory/>
   - 参考点：短期聊天历史受 token limit 约束，超限后可 flush/summarize 到长期记忆。
4. Letta/MemGPT Memory
   - Memory blocks：<https://docs.letta.com/guides/core-concepts/memory/memory-blocks/>
   - Archival memory：<https://docs.letta.com/guides/core-concepts/memory/archival-memory/>
   - 参考点：总是注入上下文的 core memory 与按需检索的 archival memory 分层。

## 当前项目映射

| 成熟项目概念 | 本项目当前实现 | 对齐状态 |
| --- | --- | --- |
| Thread/session scoped short-term memory | `ConversationLedgerStore` + `ConversationSessionStore`，每个 conversation/session 独立 ledger 和 active session pointer | 已对齐 |
| Session history source of truth | `data/conversation_ledgers/<conversation_id>/messages.jsonl` 和 `conversation.md` | 已对齐 |
| Context window assembly | `LedgerContextAssembler` 从 active ledger、引用窗口、文件/链接索引和 memory 文件组装 prompt | 已对齐 |
| Reset / new session | reset 命令只切换 session，不删除微信原始对话；新 session 默认不读取旧 session | 已对齐 |
| Core memory / always-visible memory block | `memory/summary.md`、`preferences.json`、`entities.json` 注入 `Long-term memory` section | 已补齐第一版 |

## 需要靠拢的差距

1. 长期记忆仍是确定性抽取，不是成熟项目里的 LLM-assisted memory extraction。
   - 当前优点：可测试、稳定、不依赖真实 API。
   - 后续靠拢：给 `MemoryMaintainer` 增加可插拔 summarizer/extractor，让模型只更新结构化 memory patch。
2. 归档记忆没有向量/语义检索。
   - 当前完整信息仍可从 ledger、文件中间层、annotations 追溯。
   - 后续靠拢：为 ledger text block、文件解析块、网页注释建立 SQLite FTS 或向量索引，并暴露为 tool。
3. 文件/网页/语音处理链已经有索引和中间层，但还不是完整的 tool-first agent memory 系统。
   - 当前路径适合小范围测试。
   - 后续靠拢：工具结果必须产生标准 artifact manifest，再由 maintainer/assembler 引用。
4. UI 捕获不是生产级消息源。
   - 当前策略：排除 Chrome/sidebar/终端干扰，允许手动绑定目标微信窗口，小范围自用优先稳定。
   - 后续靠拢：接真实后端消息事件源，UI/OCR 只保留为诊断或补充工具。

## 当前推荐架构结论

继续保持“ledger 是唯一对话事实源，memory 是 ledger 派生物”的架构，不恢复旧 `context_store` 消息历史。

对话上下文按以下顺序进入模型：

1. 当前 conversation + current session 的 active ledger entries。
2. 当前消息显式 quote 命中的跨 session 邻近窗口。
3. 当前 session 的 core memory：summary/preferences/entities。
4. 相关文件、网页、附件 artifact refs。
5. 在 token budget 内保留最近消息；超出预算先裁剪 recent，不裁剪 quote/files/memory。

这个形态与 LangGraph/OpenAI Agents/LlamaIndex/Letta 的共同原则一致：会话历史可追溯，短期上下文受预算管理，长期记忆独立维护，归档信息按需读取。
