# 实机稳定性核验记录

更新时间：2026-06-30

## 当前结论

### 2026-06-30 补充：通道与输入输出边界

1. 通道清理已经从“前端隐藏”升级为“删除通道注册”：
   - sidebar 的“清除通道”会删除 `data/conversation_channels/<conversation_id>/` 和 `conversation_channels/index.json` 中的注册项。
   - 不删除 `conversation_ledgers`、`file_workspace`、`conversation_sessions`，因此对话文件、文件中间层和会话证据仍可回放。
   - “清理隐藏通道”只清理被可见策略判定为旧探测/乱码/工具窗口的隐藏注册。
2. 后端事件端口已经成为结构化输入入口：
   - `POST /api/backend-events` 与 `append-backend-event --history-json/--history-file` 会写入 `backend_events.jsonl`。
   - `run-agent` / `poll-backend-events` 读取该事件流后自动创建通道、写入对应 `conversation_ledgers/<conversation_id>/conversation.md`。
   - 新事件可携带 `history`，历史消息会先按顺序写入对话文件并标记 `context_only`，只补上下文，不触发回复。
3. 窗口诊断已经稳定到“绑定/可见微信窗口优先”：
   - sidebar 的微信诊断不会因为 PowerShell、浏览器或鼠标焦点变化就把目标改成前台噪声窗口。
   - sidebar 浮窗定位优先贴合 `window_bindings.json` 中的活跃微信绑定窗口，无绑定时再退回可见微信窗口。
4. 发送边界仍需明确：
   - 当前真实发送驱动 `windows_guarded` 是前台粘贴并回车，必须目标微信聊天窗口处于前台且标题匹配。
   - 它不能实现真正后台多会话并发发送。
   - 后台多路接收已经走后端事件总线；后台多路发送需要后续新增非前台发送桥或微信后端发送驱动。

1. 微信窗口发现与 guarded 发送探测仍可用。
   - 真实发送必须继续依赖 `windows_guarded` 的前台窗口守卫。
   - 发送前仍应通过确认队列或自动发送开关审计。
2. 页面 OCR 输入链路已废弃。
   - `ocr-snapshot`、`poll-ocr-window`、`ocr-window-diagnose` 只返回 `deprecated`。
   - 这些命令不会截图、不会 OCR、不会写 ledger、不会入发送队列。
3. OCR 只保留为文件层工具。
   - 图片、扫描件、文档内图片等素材由文件中间层和 `vision.ocr`/`ocr-image` 调用。
   - 不再把微信页面截图 OCR 成对话快照。
4. 页面/窗口抓取只保留纯抓取。
   - `wechat-capture` 可以用于手工取证、窗口验证或后续后台抓取调试。
   - 抓到的图片不会自动进入对话上下文。

## 输入路径边界

稳定运行优先级：

1. 后端事件源：`poll-backend-events`。
2. 文件 watcher：`scan-backend-files` 产生事件后交给后端事件源。
3. 文本快照：`poll-snapshot`，仅用于明确格式的文本快照。
4. 剪贴板/UIA：`poll-clipboard`、`wechat-snapshot`，只作为只读调试或过渡输入。
5. 纯窗口抓取：`wechat-capture`，只取图，不 OCR，不写 ledger。

禁止路径：

- 不使用 OCR 阅读微信页面。
- 不使用 OCR 截图推断消息、文件卡片或语音转文字。
- 不把页面 OCR 结果写入 `conversation_ledgers`。
