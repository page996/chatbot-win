## 稳定性结论

后端事件源稳定：

- `append-backend-event` -> `poll-backend-events` 单轮可完成：
  - channel auto-register
  - ledger 写入
  - session memory 更新
  - reply candidate 生成
  - dry-run/confirm/auto send gate 处理

前台 OCR 源可作为小范围自助输入源，但不是成熟后台消息源：

- 窗口句柄可绑定。
- 截图 OCR 可用。
- 当截图只看到左侧截断预览、文件预览或错误聊天标题时，系统会拒绝写 ledger。
- 已绑定窗口失效时，系统会返回 `bound_window_unavailable`，不会退回抓其他窗口。

## 新增守卫

1. `chat_title_not_visible`
   - 指定 `--chat-title PAGE` 时，OCR 截图中必须能看到 `PAGE`。
   - 否则拒绝写入对话文件。
2. `bound_window_unavailable`
   - 已绑定窗口消失或无法解析时，拒绝 fallback 到其他微信窗口。
   - 绑定状态会标记为 `stale`。
3. `ocr-window-diagnose`
   - 只截图、OCR、解析和输出 readiness。
   - 不调用 MessageProcessor。
   - 不写 ledger。
   - 不生成发送队列。

## 建议操作边界

稳定运行优先级：

1. 首选后端事件/文件 watcher。
2. OCR 前台源只在微信 PAGE 聊天窗口保持前台、聊天标题可见、右侧消息气泡完整可见时使用。
3. 如果诊断返回 `blocked_by_ocr_parse`，不要自动发送；需要调整微信画面，让完整消息气泡出现在右侧聊天 pane。
4. 如果诊断返回 `blocked_by_window`，重新将目标微信聊天置前或重新绑定。
