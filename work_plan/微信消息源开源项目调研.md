# 微信消息源开源项目调研

更新时间：2026-07-01

## 结论

当前项目用 UIA、剪贴板、窗口快照、文件目录监听去拼接对话文件，本质上仍然是证据拼接，不能稳定解决长时间、高速、多消息类型对话里的消息顺序、撤回、引用、语音、图片和文件归属问题。

要显著降低猜测成分，公开项目里主要有三条路线：

1. PC Hook / SDK 实时消息流：直接从微信客户端事件里拿消息，最接近“消息队列”。
2. 微信本地 WCDB / 数据库读取：读取本地消息表、会话表、媒体索引，适合补全历史、严格排序、引用、撤回和多媒体定位。
3. 文件存储证据索引：监听 `xwechat_files` 只能证明“某个文件/媒体出现过”，适合作为附件本体解析层，不能单独作为真实对话流。

推荐下一阶段不要继续扩展 OCR 页面路径，也不要把文件监听直接写进最终对话文件。应增加一个 `MessageSourceAdapter` 抽象，允许接入 WeChatFerry 或 WeFlow/WeTrace 类 WCDB 消息源，再把文件监听降级为附件证据源。

## 推荐阅读顺序

### 1. WeChatFerry

链接：

- https://github.com/lich0821/WeChatFerry
- https://github.com/wechatferry/wechatferry
- Python 客户端文档入口：https://wechatferry.readthedocs.io/

公开 README 声明能力：

- 获取消息类型
- 获取联系人
- 获取可查询数据库
- 获取数据库表
- 获取语音消息
- 发送文本、图片、文件、卡片、XML、GIF
- 开启接收消息

对本项目的意义：

- 这是最接近“实时消息队列”的路线。
- 如果能在你的微信版本上稳定运行，它可以作为主消息源，替代当前 UI 猜测。
- 它天然更适合并发会话，因为消息事件里通常带 talker / room id / sender / type / content 等字段。

主要风险：

- PC Hook 路线强依赖微信版本，升级后可能失效。
- 可能触发安全软件或微信风控。
- 与微信用户协议、账号安全存在不确定性。
- 如果接入，建议隔离为可选插件，不要写死进核心上下文系统。

适配建议：

- 新增 `WcfMessageSourceAdapter`。
- 将收到的实时消息转换成统一的 `RawMessageEvent`。
- 只让 adapter 输出结构化事件，不让 adapter 直接写 ledger。
- 发送侧也可以先保留当前 guarded sender，等接收稳定后再考虑用 WCF 发送。

### 2. WeFlow

链接：

- https://github.com/hicccc77/WeFlow
- HTTP API 文档：https://github.com/hicccc77/WeFlow/blob/main/docs/HTTP-API.md

公开 README 声明能力：

- 完全本地的微信实时聊天记录查看、分析与导出工具。
- 支持微信 4.0 及以上。
- 实时刷新最新消息，无需生成解密中间数据库。
- 支持消息防撤回。
- 支持图片、视频、实况预览和解密。
- 提供本地 HTTP API，默认端口 `5031`。

仓库结构里值得重点阅读：

- `electron/services/wcdbCore.ts`
- `electron/services/wcdbService.ts`
- `electron/services/messageCacheService.ts`
- `electron/services/messagePushService.ts`
- `electron/services/apiMessageMapping.ts`
- `electron/services/export/media/attachmentResolver.ts`
- `electron/services/voiceTranscribeService.ts`
- `resources/wcdb/win32/x64/WCDB.dll`
- `resources/wcdb/win32/x64/wcdb_api.dll`

对本项目的意义：

- 这条路线最贴近你当前微信 4.x 的 `xwechat_files/db_storage` 目录结构。
- 它的 HTTP API 可能可以作为外部只读消息源，让本项目不直接处理数据库细节。
- 对“长对话顺序、历史补齐、媒体定位、撤回”比文件监听可靠得多。

主要风险：

- 项目涉及本地数据库读取、媒体解密，合规和账号安全需你自行评估。
- API 是否包含足够实时的增量接口，需要实际启动后核查。
- 我们应优先把它作为外部进程/服务读取，不建议直接复制其内部解密代码。

适配建议：

- 优先部署 WeFlow 成品或源码，开启本地 HTTP API。
- 本项目新增 `WeFlowHttpMessageSourceAdapter`。
- adapter 轮询或订阅 WeFlow API，按 `seq/create_time/msg_id/talker` 拉取增量消息。
- 将 WeFlow 作为主对话 ledger 来源，把本项目的文件中间层用于二次解析附件内容。

已读到的关键设计：

- 它把实时 WCDB 读取和解密后 sqlite 快照分开，实时 UI 默认读 WCDB realtime，解密 sqlite 更像兼容桥。
- 文档和代码都围绕微信 4.x 的 `db_storage` 做了专门适配。

对本项目的意义：

- 很适合作为 WCDB 实时读取的参考架构。
- 它的 FastAPI 后端结构和本项目 Python 栈更接近，适配成本可能低于 WeFlow。
- 如果它的接口足够完整，可以做外部服务适配，而不是直接整合敏感逻辑。

主要风险：

- 同样涉及数据库密钥和媒体读取，合规风险需用户自行评估。
- 依赖原生 DLL 和微信版本。
- 直接引入源码会把边界变复杂，建议进程级隔离。

适配建议：

- 先跑它自己的服务，看是否能稳定列出会话、消息、媒体和最新增量。
- 本项目新增 `WeChatDataAnalysisHttpAdapter` 或只读 `WCDBRealtimeBridge`。
- 本项目不要直接保存密钥；只消费外部服务返回的本地结构化消息。

## 对本项目的架构建议

### 新增统一消息源接口

建议把当前消息输入改造成：

```text
MessageSourceAdapter
  -> RawMessageEvent
  -> EvidenceJournal
  -> CanonicalLedgerBuilder
  -> conversation.md / messages.jsonl
  -> LLM context view
```

各消息源只负责提供事实，不直接写最终 ledger：

- `WcfMessageSourceAdapter`：实时 Hook/SDK 来源。
- `WeFlowHttpMessageSourceAdapter`：WeFlow HTTP API 来源。
- `WeChatDataAnalysisHttpAdapter`：WeChatDataAnalysis API 来源。
- `WindowsUiSnapshotAdapter`：仅保留为人工兜底。
- `WechatStorageEvidenceAdapter`：只提供附件本体证据，不单独制造消息。

### 统一事件字段

最小字段建议：

- `source`
- `source_event_id`
- `conversation_key`
- `talker_id`
- `sender_id`
- `sender_name`
- `is_self`
- `message_type`
- `content_text`
- `raw_content`
- `create_time`
- `sort_key`
- `server_id`
- `local_id`
- `quote`
- `recall`
- `attachments`
- `provenance`
- `confidence`

其中 `sort_key` 优先级：

1. 数据库 `sort_seq` / `seq`
2. 数据库 `create_time + local_id`
3. Hook 事件到达时间 + 上游 msgid
4. UI/文件观察时间，仅低置信度

### xwechat_files 的正确位置

你找到的 `xwechat_files` 路径应该纳入项目，但定位应是“附件证据源”，不是“对话消息源”。

它能解决：

- 文件本体是否已下载。
- 图片、语音、视频、文档的原始文件定位。
- 文件解析、OCR、ASR、LibreOffice 中间层处理。

它不能单独解决：

- 谁发的。
- 发给哪个会话。
- 在哪条消息位置。
- 与引用/撤回的关系。
- 多个同名文件或同时间文件的精确归属。

因此建议流程是：

```text
数据库/Hook 消息事件发现附件消息
  -> 得到 talker、msgid、时间、文件名、md5、server_id 等线索
  -> xwechat_files 索引器查找本体
  -> 高置信匹配后 copy 到 file_workspace
  -> 文件解析结果回填 ledger
```

### 优先级

1. 先部署或阅读 WeFlow，确认 HTTP API 能否给出实时消息和媒体定位。
2. 同时阅读 WeChatFerry，确认你的微信版本是否支持实时接收和获取语音/文件消息。
3. 若 WeFlow API 足够好，优先做 WeFlow HTTP 适配器，因为它与微信 4.x / `xwechat_files` 更贴近。
4. 若 WeChatFerry 稳定，则它可以作为实时消息主输入，WeFlow/WCDB 作为历史补齐与校验。
5. 保留当前 UI 捕获为兜底，不再作为主路径扩展。

## 需要你重点核查的问题

阅读或部署上游项目时，请优先核查：

1. 是否支持你当前登录微信版本。
2. 是否能输出增量消息，而不只是全量导出。
3. 单条消息是否包含稳定的 `talker/session`、`sender`、`create_time`、`seq/msgid`。
4. 是否支持群聊真实发送者。
5. 是否支持撤回消息事件或撤回后的状态。
6. 是否支持引用消息结构。
7. 是否能定位语音、图片、文件原始路径。
8. 是否能以本地 HTTP API 或 SDK 被我们的 Python 项目消费。
9. 是否需要关闭/重启微信。
10. 是否会引发账号安全提示或安全软件告警。

## 推荐落地方案

最稳妥的下一步不是在本项目里实现数据库解密或 Hook，而是做一个外部消息源适配层：

1. 你先本地部署 WeFlow 或 WeChatFerry。
2. 本项目只连接 `127.0.0.1` API / SDK。
3. 本项目消费结构化消息事件。
4. 本项目负责 conversation ledger、记忆、工具调用、文件中间层、回复队列。
5. 上游项目负责微信消息源细节。

这样能把高风险、强版本绑定的部分隔离出去，也方便你以后替换消息源。

## 本项目已新增的 Hook 采集器接入边界

本项目已经补出主项目侧的 Hook 消息入口，但没有把底层 DLL 注入、偏移定位、反调试、内存扫描等实现写进主流程。当前边界是：

```text
自制 Hook 采集器 / WeChatFerry / WeFlow / 其他外部消息源
  -> 输出 JSONL
  -> import-hook-events
  -> backend_events.jsonl
  -> BackendEventJsonlDriver
  -> AttachmentPipeline / FileWorkspace
  -> ConversationLedgerStore
  -> LedgerContextAssembler / MemoryMaintainer
  -> 回复队列
```

新增入口：

```bash
python -m app.personal_wechat_bot.main --data-dir data import-hook-events \
  --hook-event-file data/hook_events.jsonl \
  --backend-event-file data/backend_events.jsonl
```

也可以让 agent 每轮启动前自动导入：

```bash
python -m app.personal_wechat_bot.main --data-dir data run-agent \
  --hook-event-file data/hook_events.jsonl \
  --backend-event-file data/backend_events.jsonl
```

### Hook JSONL 协议

外部采集器每行输出一个 JSON 对象。推荐最小文本消息：

```json
{
  "talker": "wxid_xxx",
  "talker_name": "PAGE",
  "sender_id": "wxid_xxx",
  "sender_name": "PAGE",
  "msgid": "123456",
  "sort_seq": "1719900000001",
  "create_time": "2026-07-01T12:00:00+08:00",
  "text": "这是一条消息"
}
```

群聊消息：

```json
{
  "talker": "123456789@chatroom",
  "talker_name": "测试群",
  "sender_id": "wxid_member",
  "sender_name": "群成员昵称",
  "msgid": "123457",
  "sort_seq": "1719900000002",
  "text": "@机器人 请处理这个文件"
}
```

文件或图片消息：

```json
{
  "talker": "wxid_xxx",
  "talker_name": "PAGE",
  "sender_name": "PAGE",
  "msgid": "123458",
  "text": "请读这个文件",
  "attachments": [
    {
      "path": "E:/aab通讯文件存储/xwechat_files/.../report.pdf",
      "name": "report.pdf",
      "kind": "file",
      "size": 123456,
      "md5": "optional"
    }
  ]
}
```

语音消息：

```json
{
  "talker": "wxid_xxx",
  "talker_name": "PAGE",
  "sender_name": "PAGE",
  "msgid": "123459",
  "message_type": "voice",
  "voice": {
    "status": "pending",
    "duration": "6s",
    "audio_path": "E:/.../voice.m4a",
    "audio_name": "voice.m4a"
  }
}
```

如果 Hook 层已经拿到微信自带转文字结果：

```json
{
  "talker": "wxid_xxx",
  "talker_name": "PAGE",
  "sender_name": "PAGE",
  "msgid": "123460",
  "message_type": "voice",
  "voice": {
    "status": "transcribed",
    "text": "语音转出来的文字",
    "duration": "6s"
  }
}
```

引用消息：

```json
{
  "talker": "wxid_xxx",
  "talker_name": "PAGE",
  "sender_name": "PAGE",
  "msgid": "123461",
  "text": "继续这条",
  "quote": {
    "message_id": "optional-normalized-message-id",
    "text": "被引用的正文",
    "sender_name": "PAGE",
    "received_at": "2026-07-01T11:59:00+08:00"
  }
}
```

撤回消息：

```json
{
  "event_type": "recall",
  "talker": "wxid_xxx",
  "talker_name": "PAGE",
  "msgid": "recall-event-1",
  "target_raw_id": "hook:message:wxid_xxx:123456",
  "reason": "wechat_recall"
}
```

字段兼容说明：

- `talker`、`talker_id`、`roomid`、`room_id`、`conversation_key` 都会被识别为稳定会话键。
- `talker_name`、`room_name`、`display_name`、`chat_title` 会被识别为显示标题。
- `sender_id`、`from_wxid`、`sender_wechat_id` 会被识别为发送人 ID。
- `msgid`、`msg_id`、`message_id`、`server_id`、`local_id`、`sort_seq` 会进入元数据，并参与稳定 raw id。
- 有 `conversation_key` 后，聊天标题变化不会导致 ledger 分裂。

### 当前实现范围

已实现：

- Hook JSONL 导入器。
- Hook 字段归一化。
- 增量 offset 状态文件，避免重复导入旧行。
- 附件进入文件中间层。
- 语音 pending / transcribed 进入既有语音路径。
- 引用元数据进入 ledger quote。
- 撤回事件可以标记已写入 ledger 的目标消息为 recalled。
- `run-agent --hook-event-file` 自动先导入 Hook 事件再处理。

未实现：

- 底层微信函数 Hook。
- DLL 注入或偏移定位。
- 数据库密钥提取。
